# Test-Driven Development (TDD) Skill\n\n## 개요\n\nRED-GREEN-REFACTOR 사이클을 따르는 테스트 우선 개발 방식입니다.\n\n## 철학\n\n> 테스트를 먼저 작성하면, 코드의 설계가 자동으로 개선됩니다.\n\n- **테스트 = 명세서**: 테스트는 코드가 어떻게 동작해야 하는지 명시합니다\n- **빠른 피드백**: 변경 사항이 즉시 검증됩니다\n- **자신감 있는 리팩토링**: 테스트가 있으면 안전하게 코드를 개선할 수 있습니다\n- **문서화**: 테스트는 코드 사용법을 보여주는 문서입니다\n\n## RED-GREEN-REFACTOR 사이클\n\n### 1단계: RED (실패하는 테스트 작성)\n\n```javascript\n// test/calculator.test.js\ndescribe('Calculator', () => {\n  it('should add two numbers', () => {\n    const result = add(2, 3);\n    expect(result).toBe(5);\n  });\n});\n```\n\n**상태**: ❌ 테스트 실패 (함수가 아직 없음)\n\n### 2단계: GREEN (최소한의 코드로 테스트 통과)\n\n```javascript\n// src/calculator.js\nfunction add(a, b) {\n  return a + b;\n}\n```\n\n**상태**: ✅ 테스트 통과\n\n### 3단계: REFACTOR (코드 개선)\n\n```javascript\n// 더 나은 구조로 리팩토링\nclass Calculator {\n  add(a, b) {\n    return a + b;\n  }\n}\n```\n\n**상태**: ✅ 테스트 여전히 통과\n\n## 구현 가이드\n\n### 테스트 작성 규칙\n\n1. **명확한 이름**: 테스트가 무엇을 검증하는지 명확해야 합니다\n   ```javascript\n   // ❌ 나쁜 예\n   it('works', () => { ... });\n   \n   // ✅ 좋은 예\n   it('should return sum when adding two positive numbers', () => { ... });\n   ```\n\n2. **AAA 패턴**: Arrange-Act-Assert\n   ```javascript\n   it('should calculate total price with tax', () => {\n     // Arrange: 테스트 데이터 준비\n     const price = 100;\n     const taxRate = 0.1;\n     \n     // Act: 함수 실행\n     const total = calculateTotal(price, taxRate);\n     \n     // Assert: 결과 검증\n     expect(total).toBe(110);\n   });\n   ```\n\n3. **한 가지만 테스트**: 각 테스트는 하나의 동작만 검증합니다\n   ```javascript\n   // ❌ 나쁜 예: 여러 것을 테스트\n   it('should validate user', () => {\n     expect(validateEmail(email)).toBe(true);\n     expect(validatePassword(password)).toBe(true);\n   });\n   \n   // ✅ 좋은 예: 한 가지만 테스트\n   it('should validate email format', () => {\n     expect(validateEmail('test@example.com')).toBe(true);\n   });\n   ```\n\n### 테스트 커버리지\n\n**목표**: 80% 이상의 코드 커버리지\n\n```bash\n# 커버리지 확인\nnpm run test:coverage\n```\n\n**커버리지 유형**:\n- **Line Coverage**: 실행된 코드 라인의 비율\n- **Branch Coverage**: 실행된 분기(if/else)의 비율\n- **Function Coverage**: 실행된 함수의 비율\n\n## 안티패턴\n\n### 1. 테스트 없이 구현\n```javascript\n// ❌ 나쁜 예\nfunction complexLogic() {\n  // 복잡한 로직\n  // 테스트 없음\n}\n```\n\n### 2. 테스트가 너무 복잡함\n```javascript\n// ❌ 나쁜 예\nit('should do something', () => {\n  // 100줄의 복잡한 설정 코드\n  // 실제 테스트는 1줄\n});\n```\n\n### 3. 테스트 중복\n```javascript\n// ❌ 나쁜 예\nit('should add 1 + 1', () => expect(add(1, 1)).toBe(2));\nit('should add 1 + 1 again', () => expect(add(1, 1)).toBe(2));\n```\n\n### 4. 테스트 간 의존성\n```javascript\n// ❌ 나쁜 예\nlet user;\nit('should create user', () => {\n  user = createUser('John');\n  expect(user).toBeDefined();\n});\nit('should get user', () => {\n  // 이전 테스트에 의존\n  expect(getUser(user.id)).toBeDefined();\n});\n```\n\n## 체크리스트\n\n- [ ] 각 기능에 대한 테스트 작성\n- [ ] RED-GREEN-REFACTOR 사이클 준수\n- [ ] 테스트 이름이 명확함\n- [ ] AAA 패턴 사용\n- [ ] 80% 이상의 커버리지\n- [ ] 엣지 케이스 테스트 포함\n- [ ] 테스트 간 의존성 없음\n\n## 예제\n\n### 사용자 인증 테스트\n\n```javascript\ndescribe('Authentication', () => {\n  describe('validatePassword', () => {\n    it('should return true for valid password', () => {\n      const password = 'SecurePass123!';\n      expect(validatePassword(password)).toBe(true);\n    });\n\n    it('should return false for password less than 8 characters', () => {\n      const password = 'Short1!';\n      expect(validatePassword(password)).toBe(false);\n    });\n\n    it('should return false for password without uppercase', () => {\n      const password = 'securepass123!';\n      expect(validatePassword(password)).toBe(false);\n    });\n  });\n});\n```\n\n## 참고 자료\n\n- [Jest Documentation](https://jestjs.io/)\n- [Testing Library](https://testing-library.com/)\n- [Test Pyramid](https://martinfowler.com/bliki/TestPyramid.html)\n"
