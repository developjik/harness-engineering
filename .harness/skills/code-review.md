# Code Review Skill\n\n## 개요\n\n체계적인 코드 리뷰 프로세스를 통해 코드 품질을 보장합니다.\n\n## 리뷰 기준\n\n### 1. 기능 정확성 (Correctness)\n\n**확인 사항:**\n- 설계 문서와 일치하는가?\n- 모든 요구사항이 구현되었는가?\n- 엣지 케이스를 처리하는가?\n- 에러 처리가 적절한가?\n\n**체크리스트:**\n```\n[ ] 요구사항과 일치\n[ ] 엣지 케이스 처리\n[ ] 에러 처리 완료\n[ ] 테스트 커버리지 충분\n```\n\n### 2. 코드 품질 (Code Quality)\n\n**SOLID 원칙:**\n- **S**ingle Responsibility: 한 클래스는 한 가지 책임만\n- **O**pen/Closed: 확장에는 열려있고 수정에는 닫혀있음\n- **L**iskov Substitution: 상속 관계가 올바름\n- **I**nterface Segregation: 인터페이스가 작고 구체적\n- **D**ependency Inversion: 추상화에 의존\n\n**체크리스트:**\n```\n[ ] 함수/클래스가 한 가지 책임만 함\n[ ] 중복 코드가 없음 (DRY)\n[ ] 불필요한 코드가 없음 (YAGNI)\n[ ] 변수/함수 이름이 명확함\n[ ] 복잡도가 적절함 (Cyclomatic Complexity < 10)\n```\n\n### 3. 보안 (Security)\n\n**일반적인 취약점:**\n- SQL Injection\n- XSS (Cross-Site Scripting)\n- CSRF (Cross-Site Request Forgery)\n- 인증/인가 문제\n- 민감한 정보 노출\n\n**체크리스트:**\n```\n[ ] 입력 검증 수행\n[ ] 출력 이스케이프 처리\n[ ] 인증/인가 확인\n[ ] 민감한 정보 보호\n[ ] 의존성 취약점 확인\n```\n\n### 4. 성능 (Performance)\n\n**확인 사항:**\n- 불필요한 루프가 있는가?\n- 데이터베이스 쿼리가 최적화되어 있는가?\n- 메모리 누수가 없는가?\n- 캐싱이 적절히 사용되는가?\n\n**체크리스트:**\n```\n[ ] 알고리즘 복잡도 적절 (O(n) 이하)\n[ ] 데이터베이스 쿼리 최적화\n[ ] 불필요한 API 호출 없음\n[ ] 메모리 누수 없음\n```\n\n### 5. 테스트 (Testing)\n\n**확인 사항:**\n- 테스트 커버리지가 충분한가? (80% 이상)\n- 테스트가 명확한가?\n- 엣지 케이스 테스트가 있는가?\n- 테스트가 독립적인가?\n\n**체크리스트:**\n```\n[ ] 커버리지 80% 이상\n[ ] 테스트 이름이 명확\n[ ] 엣지 케이스 테스트 포함\n[ ] 테스트 간 의존성 없음\n[ ] 모든 테스트 통과\n```\n\n## 리뷰 프로세스\n\n### 1단계: 변경 사항 이해\n```\n1. PR 설명 읽기\n2. 관련 이슈 확인\n3. 변경된 파일 목록 확인\n```\n\n### 2단계: 코드 검토\n```\n1. 전체 흐름 파악\n2. 각 함수/클래스 검토\n3. 테스트 코드 검토\n4. 문서 검토\n```\n\n### 3단계: 피드백 작성\n```\n1. 긍정적인 피드백 먼저\n2. 개선 사항 제시\n3. 질문하기 (지시하지 않기)\n4. 명확한 예제 제공\n```\n\n### 4단계: 의사결정\n```\n✅ 승인: 모든 기준 충족\n⚠️ 조건부 승인: 경미한 개선 필요\n❌ 반려: 주요 문제 있음\n```\n\n## 피드백 작성 가이드\n\n### 좋은 피드백\n\n```\n// ✅ 구체적이고 건설적\n\"이 함수는 너무 많은 책임을 가지고 있습니다. \n검증 로직을 별도의 함수로 분리하면 테스트하기 쉬워질 것 같습니다.\n\n예:\nfunction validateUser(user) {\n  // 검증 로직\n}\n\nfunction createUser(user) {\n  validateUser(user);\n  // 생성 로직\n}\"\n```\n\n### 나쁜 피드백\n\n```\n// ❌ 모호하고 비판적\n\"이 코드는 좋지 않습니다.\"\n\"왜 이렇게 했나요?\"\n```\n\n## 리뷰 체크리스트\n\n### 필수 항목\n- [ ] 코드가 요구사항을 충족하는가?\n- [ ] 테스트가 충분한가?\n- [ ] 보안 문제가 없는가?\n- [ ] 성능 문제가 없는가?\n\n### 권장 항목\n- [ ] 코드 스타일이 일관적인가?\n- [ ] 문서가 최신인가?\n- [ ] 의존성이 최소화되어 있는가?\n- [ ] 에러 처리가 적절한가?\n\n## 예제\n\n### 나쁜 코드\n\n```javascript\nfunction processData(data) {\n  let result = [];\n  for (let i = 0; i < data.length; i++) {\n    if (data[i].active) {\n      let user = data[i];\n      let name = user.firstName + ' ' + user.lastName;\n      let email = user.email;\n      result.push({\n        name: name,\n        email: email\n      });\n    }\n  }\n  return result;\n}\n```\n\n### 개선된 코드\n\n```javascript\nfunction getActiveUsers(users) {\n  return users\n    .filter(user => user.active)\n    .map(user => ({\n      name: `${user.firstName} ${user.lastName}`,\n      email: user.email\n    }));\n}\n```\n\n### 리뷰 피드백\n\n```\n✅ 좋은 점:\n- 함수 이름이 명확합니다\n- 한 가지 책임만 합니다\n\n💡 개선 사항:\n1. 함수형 프로그래밍 스타일 사용\n2. 변수 이름이 더 명확할 수 있습니다\n3. 테스트 케이스를 추가하세요\n\n예:\nfunction getActiveUsers(users) {\n  return users\n    .filter(user => user.active)\n    .map(formatUserInfo);\n}\n\nfunction formatUserInfo(user) {\n  return {\n    name: `${user.firstName} ${user.lastName}`,\n    email: user.email\n  };\n}\n```\n\n## 참고 자료\n\n- [Google Code Review Guidelines](https://google.github.io/eng-practices/review/)\n- [Code Review Best Practices](https://smartbear.com/learn/code-review/best-practices/)\n- [SOLID Principles](https://en.wikipedia.org/wiki/SOLID)\n"
