## 자동화 레벨 (L0-L4)

```
┌───────────────────────────────────────────────────────────────────────┐
│                      자동화 레벨 시스템                                │
├────┬───────────┬──────────────────────────────────────────────────────┤
│레벨│   이름    │                    설명                               │
├────┼───────────┼──────────────────────────────────────────────────────┤
│ L0 │ Manual    │ 모든 전환에 승인 필요 ← 초보자, 중요 프로젝트         │
│ L1 │ Guided    │ 중요 전환만 승인 ← 학습 단계                          │
│ L2 │ Semi-Auto │ 불확실할 때만 승인 (기본값) ← 일반 사용자             │
│ L3 │ Auto      │ 품질 게이트만 통과하면 자동 ← 숙련자                  │
│ L4 │ Full-Auto │ 완전 자동 ← 매우 숙련된 사용자                        │
└────┴───────────┴──────────────────────────────────────────────────────┘
```

## Context Rot 방지 시스템

긴 세션에서 발생하는 컨텍스트 품질 저하를 감지하고 관리합니다.

```
점수 계산:
  score = (토큰비율 × 0.4) + (작업비율 × 0.3) + (시간비율 × 0.3)

등급:
  < 0.5: healthy (건강)
  0.5-0.7: caution (주의)
  >= 0.7: rot (서브에이전트 권장)
```

## 핵심 설계 원칙

1. **SSOT (Single Source of Truth)**: `docs/specs/<feature-slug>/`에 모든 산출물 집중
2. **프롬프트 중심 오케스트레이션**: 복잡한 런타임 코드 대신 스킬/에이전트 지침으로 제어
3. **얕은 훅 자동화**: Bash 훅으로 보안, 백업, 로깅을 가볍게 수행
4. **문서/실행 분리**: 산출물은 `docs/specs/`, 런타임은 `.harness/`에 분리

## 세 계층 구조

| 계층 | 위치 | 역할 |
|:-----|:-----|:-----|
| **프롬프트 계층** | `skills/`, `agents/` | 실행 절차와 사고 방식 정의 |
| **운영 자동화 계층** | `hooks.json`, `hooks/` | 보안, 백업, 로깅, 상태 추적 |
| **산출물/상태 계층** | `docs/specs/`, `.harness/` | 기능별 문서와 런타임 상태 |

## 데이터 흐름 요약

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          데이터 흐름 요약                                    │
└─────────────────────────────────────────────────────────────────────────────┘

    입력                           처리                         출력
    ────                           ────                         ────

    사용자 요청                ┌─────────────┐
         │                    │  /clarify   │────────▶ clarify.md
         │                    └──────┬──────┘
         │                           │
         │                    ┌──────▼──────┐
         │                    │   /plan     │────────▶ plan.md
         │                    └──────┬──────┘
         │                           │
         │                    ┌──────▼──────┐
         │                    │  /design    │────────▶ design.md
         │                    └──────┬──────┘
         │                           │
         │                    ┌──────▼──────┐
         └───────────────────▶│ /implement  │────────▶ 코드 + 테스트
                              └──────┬──────┘
                                     │
                              ┌──────▼──────┐
                              │   /check    │────────▶ 리뷰 결과
                              └──────┬──────┘
                                     │
                              ┌──────▼──────┐
                              │  /wrapup    │────────▶ wrapup.md
                              └─────────────┘
```

### SSOT (Single Source of Truth)

```
    docs/specs/<feature-slug>/
    │
    ├── clarify.md  ◀── 모호성 해소 결과 (Plan 입력)
    ├── plan.md     ◀── 요구사항 정의 (Design 입력)
    ├── design.md   ◀── 기술 설계 (Implement 입력)
    └── wrapup.md   ◀── 완료 문서 (아카이브)

    • 각 단계는 이전 단계의 산출물만 참조
    • 산출물은 한번 생성되면 불변 (수정 시 새 버전)
    • 모든 의사결정은 산출물에 기록
```
