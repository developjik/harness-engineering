# Brainstorm Skill

사용자의 프로젝트 설명을 받아 요구사항을 정제하고 설계 아이디어를 도출합니다.

## 1단계: 요구사항 정제

다음 질문들을 통해 요구사항을 명확히 합니다:

- 이 프로젝트의 주요 목표는 무엇인가요?
- 성공의 기준은 무엇인가요? (성능, 사용자 수, 기능 등)
- 제약 조건이 있나요? (시간, 리소스, 기술 스택, 예산)
- 기존 시스템과의 통합이 필요한가요?
- 예상 사용자 수와 트래픽은?
- 보안 및 규정 준수 요구사항이 있나요?

## 2단계: 아이디어 도출

최소 3가지 설계 옵션을 제시합니다:

**각 옵션마다**:
- 아키텍처 패턴
- 기술 스택
- 장점
- 단점
- 예상 개발 시간

## 3단계: 평가 및 선택

각 옵션을 다음 기준으로 평가합니다:
- 복잡도
- 개발 시간
- 유지보수 난이도
- 확장 가능성
- 팀의 경험도

## 4단계: 설계 문서 작성

선택된 설계에 대해 다음을 포함한 상세 문서를 작성합니다:

```markdown
# [프로젝트명] 설계 문서

## 1. 요구사항 정리
- 기능 요구사항
- 비기능 요구사항
- 제약 조건

## 2. 아키텍처 개요
- 시스템 다이어그램
- 주요 컴포넌트
- 데이터 흐름

## 3. 기술 스택
- Backend
- Frontend
- Database
- 각 선택의 이유

## 4. 데이터 모델
- 주요 엔티티
- 관계도
- 스키마 설계

## 5. API 설계
- 주요 엔드포인트
- 요청/응답 형식
- 에러 처리

## 6. 보안 전략
- 인증 방식
- 인가 방식
- 데이터 보호

## 7. 성능 계획
- 예상 트래픽
- 확장성 전략
- 캐싱 전략

## 8. 위험 요소 및 완화 전략
- 기술적 위험
- 비즈니스 위험
- 완화 방안

## 9. 구현 단계별 계획
- Phase 1: [마일스톤]
- Phase 2: [마일스톤]
- Phase 3: [마일스톤]

## 10. 성공 기준
- 기능 완성도
- 성능 지표
- 품질 지표
```

## 출력 형식

설계 완료 후 다음과 같이 정리합니다:

```
✅ 요구사항 분석 완료
- [요구사항 1]
- [요구사항 2]
- ...

🎯 추천 설계
[선택한 옵션 설명]

📋 설계 문서
[마크다운 형식의 상세 문서]

❓ 다음 단계
Engineer 에이전트가 설계 문서를 기반으로 구현을 시작할 준비가 되었습니다.
```

## 체크리스트

설계 문서 작성 후 다음을 확인하세요:

- [ ] 사용자의 모든 요구사항 이해
- [ ] 최소 3가지 설계 옵션 제시
- [ ] 각 옵션의 장단점 분석
- [ ] 선택된 설계에 대한 명확한 이유
- [ ] 구현 가능성 검증
- [ ] 설계 문서 작성 완료
- [ ] 사용자 승인 획득

## 주의사항

- 사용자의 의견을 충분히 반영하세요
- 너무 복잡한 설계는 피하세요 (YAGNI - You Aren't Gonna Need It)
- 팀의 경험도를 고려하세요
- 나중에 변경 가능한 결정과 불가능한 결정을 구분하세요
- 과도한 최적화는 피하세요
