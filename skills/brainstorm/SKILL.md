---\nname: brainstorm\ndescription: 프로젝트 요구사항을 분석하고 설계 아이디어를 도출합니다. 새로운 프로젝트나 기능을 시작할 때 사용하세요.\n---\n\n# Brainstorm Skill\n\n사용자의 프로젝트 설명을 받아 다음을 수행합니다:\n\n## 1단계: 요구사항 정제\n\n다음 질문들을 통해 요구사항을 명확히 합니다:\n- 이 프로젝트의 주요 목표는 무엇인가요?\n- 성공의 기준은 무엇인가요? (성능, 사용자 수, 기능 등)\n- 제약 조건이 있나요? (시간, 리소스, 기술 스택, 예산)\n- 기존 시스템과의 통합이 필요한가요?\n- 예상 사용자 수와 트래픽은?\n- 보안 및 규정 준수 요구사항이 있나요?\n\n## 2단계: 아이디어 도출\n\n최소 3가지 설계 옵션을 제시합니다:\n\n**각 옵션마다**:\n- 아키텍처 패턴\n- 기술 스택\n- 장점\n- 단점\n- 예상 개발 시간\n\n## 3단계: 평가 및 선택\n\n각 옵션을 다음 기준으로 평가합니다:\n- 복잡도\n- 개발 시간\n- 유지보수 난이도\n- 확장 가능성\n- 팀의 경험도\n\n## 4단계: 설계 문서 작성\n\n선택된 설계에 대해 다음을 포함한 상세 문서를 작성합니다:\n\n```markdown\n# [프로젝트명] 설계 문서\n\n## 1. 요구사항 정리\n- 기능 요구사항\n- 비기능 요구사항\n- 제약 조건\n\n## 2. 아키텍처 개요\n- 시스템 다이어그램\n- 주요 컴포넌트\n- 데이터 흐름\n\n## 3. 기술 스택\n- Backend\n- Frontend\n- Database\n- 각 선택의 이유\n\n## 4. 데이터 모델\n- 주요 엔티티\n- 관계도\n- 스키마 설계\n\n## 5. API 설계\n- 주요 엔드포인트\n- 요청/응답 형식\n- 에러 처리\n\n## 6. 보안 전략\n- 인증 방식\n- 인가 방식\n- 데이터 보호\n\n## 7. 성능 계획\n- 예상 트래픽\n- 확장성 전략\n- 캐싱 전략\n\n## 8. 위험 요소 및 완화 전략\n- 기술적 위험\n- 비즈니스 위험\n- 완화 방안\n\n## 9. 구현 단계별 계획\n- Phase 1: [마일스톤]\n- Phase 2: [마일스톤]\n- Phase 3: [마일스톤]\n\n## 10. 성공 기준\n- 기능 완성도\n- 성능 지표\n- 품질 지표\n```\n\n## 출력 형식\n\n설계 완료 후 다음과 같이 정리합니다:\n\n```\n✅ 요구사항 분석 완료\n- [요구사항 1]\n- [요구사항 2]\n- ...\n\n🎯 추천 설계\n[선택한 옵션 설명]\n\n📋 설계 문서\n[마크다운 형식의 상세 문서]\n\n❓ 다음 단계\nEngineer 에이전트가 설계 문서를 기반으로 구현을 시작할 준비가 되었습니다.\n```\n\n## 체크리스트\n\n설계 문서 작성 후 다음을 확인하세요:\n\n- [ ] 사용자의 모든 요구사항 이해\n- [ ] 최소 3가지 설계 옵션 제시\n- [ ] 각 옵션의 장단점 분석\n- [ ] 선택된 설계에 대한 명확한 이유\n- [ ] 구현 가능성 검증\n- [ ] 설계 문서 작성 완료\n- [ ] 사용자 승인 획득\n\n## 주의사항\n\n- 사용자의 의견을 충분히 반영하세요\n- 너무 복잡한 설계는 피하세요 (YAGNI - You Aren't Gonna Need It)\n- 팀의 경험도를 고려하세요\n- 나중에 변경 가능한 결정과 불가능한 결정을 구분하세요\n- 과도한 최적화는 피하세요\n"
